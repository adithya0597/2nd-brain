# Grill Report: 3-Layer Knowledge Pipeline & Activity Hooks

**Target**: Two responses proposing (1) 3-layer Capture/Compile/Query architecture and (2) vault_nodes metadata enrichment, inspired by Karpathy's self-improving wiki and user's activity hooks idea
**Date**: 2026-04-03
**Griller team**: 7 independent adversarial agents (zero shared context)

## Executive Summary

The proposal is architecturally sound but massively premature for the system's actual usage data: 22 captures in ~25 days, 19 daily notes, 98 vault files, 0 concept notes. Six of seven grillers flagged the compile job (Decision 2) as the weakest element, with an average score of 2.6/10. The only decisions with strong support are "allow unlinked captures" (D4, avg 5.6/10) and "reject activity hooks as new node type" (D7, avg 5.7/10). The 4-hour estimate is off by 3-4x (realistic: 11-14 hours). The strongest alternative path missed: close the feedback loop between the intent extractor and the classifier by persisting extraction results to `captures_log` and using them in context loading. This delivers 80% of the value in 60-90 minutes without any new architectural layer.

## Per-Lens Critiques

### Devil's Advocate
Key findings:
- 2 of 7 proposed metadata columns (`word_count`, `frontmatter_json`) already exist in `vault_nodes`
- Activity commands contradict the earlier decision to reject command prefixes (/task, /idea) as "premature for 2-5 captures/day"
- Only 22 captures in 25 days. The compile job would process a table with fewer rows than this report has paragraphs
- The "unlinked captures" concept assumes classification failures are common, but the classifier was just fixed in this session

### Feasibility Audit
Key findings:
- Realistic estimate: 8.5-14 hours (vs. proposed 4). Breakdown: schema migration (1h), capture wiring (1.5h), compile job (3-4h with LLM prompt tuning), triage integration (1.5h), activity commands (1.5h), tests (3-5h), docs (1h)
- Recommends 2-phase approach: Phase A (migration + capture + triage, 4-5hr), Phase B (compile job, 4-6hr with a 30-day data gate between them)
- The `_on_vault_write()` hook chain (7 steps per file write) will saturate the thread pool executor if the compile job creates 20+ files in one run
- Missing prerequisite: no rollback mechanism for LLM-generated vault content

### Bias Detection
Key findings:
- Authority bias on Karpathy: his compile pattern is for a personal wiki with 1000s of daily inputs. The Second Brain has 22 captures. The 50:1 scale mismatch is never addressed
- Unsourced "80%" claim ("80% of captures will flow correctly"). No measurement, no baseline, no definition of "correctly"
- Confirmation bias: user's activity hooks idea was dismissed before exploring whether it had merit at the node-connection level
- No falsifiability criteria: no metric defined for when to abandon the compile job if it produces low-quality output

### Cost-Benefit Challenge
Key findings:
- Compile job has worst ROI: ~6 hours to build for a vault with 0 concepts and 98 files. Cost of inaction is zero for 6+ months
- Real bottleneck is usage consistency (0.9 captures/day), not architecture. Most valuable 4 hours: make capture so frictionless that volume doubles
- Activity commands query a table with 22 rows. `/reminders` on 22 rows is `SELECT * FROM captures_log WHERE intent='task' AND due_date IS NOT NULL`. That's a trivial query, not a command
- 1-year TCO: ~14 hours build + 2hr/month maintenance + $50 API costs = ~38 hours + $50. Value delivered: unknown (no success metrics defined)

### Alternative Paths
Key findings:
- **Best unexplored path**: Close the feedback loop between intent extractor and classifier. Persist `ExtractionResult` fields to `captures_log`, use them in context loading. 60-90 minutes, 3 files changed, delivers 80% of value
- Cross-domain insight (journalism): "editorial triage" pattern. Don't build downstream processing until you have enough incoming material to need it. At 0.9 captures/day, manual triage via /today takes 30 seconds
- The 3-layer architecture fights the system's event-driven design. The existing post-write hooks are event-driven. A weekly batch compile job is a different paradigm
- Time-boxed 90-minute alternative: add `intent`, `extracted_project`, `due_date` to `captures_log` (10 min), persist ExtractionResult (15 min), add intent filter to search (15 min), intent-aware /today queries (20 min)

### Risk Amplification
Key findings:
- **#1 Unidentified Risk: No vault backup before compile writes.** The compile job creates/modifies vault files via LLM. No git snapshot, no staging directory, no undo. A bad compile run could touch 50 files with no rollback
- **#2: Thread pool saturation.** Compile job creating 20 stub files triggers 140 executor tasks (7 hooks x 20 files). Default ThreadPoolExecutor has ~12-16 workers. Captures during compile run hang
- **#3: No provenance tracking.** After 6 months, vault mixes human-written and LLM-generated content with no `source` marker. Cannot audit, filter, or bulk-revert machine content
- Estimate risk awareness: 1/10. A 4-hour estimate for schema migration + capture changes + LLM compile job + triage integration + 3 new commands + tests is not credible

### User Impact Assessment
Key findings:
- **Overall verdict**: "80% engineering self-indulgence, 20% genuine user improvement"
- Only 2 of 10 decisions matter to the user: D4 (unlinked captures, ~30 min) and D5 (triage in /today, ~5 min)
- The user needs to build the habit of consistent capture before compilation has anything to compile
- Activity commands (D6) provide a command-line interface to a feature the user didn't ask for. They asked for activity hooks, which were rejected
- Karpathy's pattern solves a 1000-doc problem. This vault has fewer files than a small blog

## Blind Spots Exposed

1. **Feedback loop between extractor and classifier** (flagged by Alternative Paths only): The intent extractor already runs on every capture but its output is discarded after the confirmation UI. Persisting it would make captures queryable by intent immediately, no compile layer needed.

2. **Vault backup before automated writes** (flagged by Risk Amplifier only): No mechanism to snapshot the vault before a compile job. Git commit, staging directory, or `<!-- auto-summary -->` markers would all help.

3. **Thread pool saturation** (flagged by Risk Amplifier only): The `_on_vault_write()` hook chain (7 tasks per file) combined with the compile job creating many files will exhaust the executor and block concurrent captures.

4. **Provenance tracking** (flagged by Risk Amplifier only): No `source: compile-job` frontmatter or section markers for machine-generated content. Cannot distinguish human vs. LLM content after 6 months.

5. **Usage consistency is the real bottleneck** (flagged by Cost-Benefit and User Impact): At 0.9 captures/day, the problem is input volume, not downstream architecture. Building a compile layer for 22 captures is like building a data warehouse for a spreadsheet.

## Alternative Approaches Missed

1. **Persist ExtractionResult to captures_log** (60-90 min): Add `intent`, `extracted_project`, `due_date` columns. Wire into context_loader for /today and /close. No new architecture, no compile layer, works with event-driven design.

2. **Classifier feedback loop**: Use confirmed extractions as training signal for the keyword tier. A confirmed task capture about "Second Brain project" strengthens that project's keyword associations. Tightens the existing pipeline instead of building a parallel one.

3. **Manual triage via Telegram buttons**: Instead of automated compile, add "Link to Project" and "Set Dimension" inline buttons to every capture confirmation. Let the user classify at capture time. At 0.9 captures/day, this adds ~15 seconds/day.

## Final Verdict

**APPROVE WITH MAJOR REVISIONS**

What to keep:
- **D4: Unlinked captures** (avg 5.6/10). Ship this immediately. ~30 minutes. Real UX fix.
- **D7: Reject activity hooks as separate node type** (avg 5.7/10). Correct architectural call.
- **Intent-aware extraction** (the gate fix already shipped). This is working. Build on it.

What must change before proceeding:
- **Kill the compile job (D2)**. Average 2.6/10. Premature for 98 files and 22 captures. Revisit when vault hits 500+ files and unlinked captures exceed 50.
- **Reduce metadata scope (D3)**. Don't add 7 columns. Add 1: `intent` to `captures_log`. Two proposed columns already exist. The rest have no consumers.
- **Kill activity commands (D6)**. Average 3.4/10. At 22 captures, these return empty results. Let intent data accumulate for 60+ days first.
- **Fix the estimate (D10)**. Average 2.3/10. If you build the full proposal: 11-14 hours, not 4.

What to do instead (the 90-minute path):
1. Add `intent TEXT`, `extracted_project TEXT`, `due_date TEXT` to `captures_log` (10 min)
2. Persist `ExtractionResult` in `capture.py` after extraction (15 min)
3. Allow unlinked captures with deferred classification (20 min)
4. Add intent-aware queries to `context_loader.py` for /today (20 min)
5. Use for 30 days. Measure: how many captures have intent? How many are unlinked? How many get triaged vs. ignored?
6. Then decide: does compile earn its place, or does the event-driven pipeline handle it?

Build for the 22-capture user you have. Not the 1000-capture user you imagine having in 6 months.
