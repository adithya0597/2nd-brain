# Grill Report: Graph Memory Research & Impact Analysis

**Target**: `vault/Reports/2026-03-28-graph-memory-research.md`
**Date**: 2026-03-28
**Griller team**: 7 independent adversarial agents (zero shared context)

## Executive Summary

The research document is architecturally sophisticated and demonstrates deep domain knowledge, but it is **materially premature for a 28-day-old system with 52 documents**. All 7 independent reviewers converged on the same core critique: the proposed additions solve problems the system doesn't yet have at current scale. The strongest decisions are the rejections (stay on SQLite, skip EdgeQuake, defer quarterly consolidation) and the advisory-only guardrail. The weakest are the implementation order (lowest-impact features ship first), temporal edge validity (massive complexity for near-zero current value), and the 3-tier configuration profiles (premature abstraction for a single-user tool). The most dangerous unidentified risk is **provenance erasure** -- LLM-generated summaries becoming indistinguishable from user-authored content in the knowledge graph, gradually replacing the user's authentic voice with Claude's interpretation of it.

## Per-Lens Critiques

### Devil's Advocate

**Strongest challenges:**
- **Forgetting Curve (Defensibility: 4/10)**: Ebbinghaus studied nonsense syllables in 1880. A legal document from 3 years ago hasn't "decayed" -- it's simply not currently relevant. Memory strength should be a property of the query-node relationship, not the node itself.
- **Consolidation (5/10)**: The 85% token savings argument really means the current context loading is naive. The fix is smarter selective retrieval, not a summarization pipeline.
- **Profiles (4/10)**: A user wanting aggressive graduation but conservative decay cannot express that. The profile conflates independent concerns under a single dial.

**Most defensible**: Advisory-only forgetting (8/10) and stay on SQLite (6/10).

### Feasibility Audit

**Biggest underestimates:**
- **Temporal Edges**: Estimated 3-4 days, realistic 7-10 days. The `ON CONFLICT(source, target, type)` clause conflicts with the new partial unique index. `bulk_upsert_edges()` needs rewriting. Community detection's hard-delete pattern must become soft-close.
- **Concept Graduation**: Missing prerequisite -- captures_log rows have no embeddings. A new embedding pipeline for captures is required (not mentioned in the plan).
- **Overall timeline**: 18 days estimated, realistic 20-28 days before counting integration testing.

**Most feasible**: Intent-Aware Routing (9/10 as code change, 3/10 as value-generating decision). The search_log table alone is the highest-ROI 1-day investment.

### Bias Detection

**Systemic patterns (flagged across all 12 decisions):**
1. **Build bias over buy**: Consistently favors native solutions without measuring whether they match proven tools' quality.
2. **Precision theater**: Specific thresholds (eps=0.35, cosine >= 0.75, 24-hour cooldown) create an impression of empirical rigor while being unvalidated against actual vault data.
3. **Foreclosed design spaces**: The most consequential choices (auto-archival, narrative vs. structured, EdgeQuake algorithms) are dismissed with caps-lock absolutes ("NEVER"), preventing future reconsideration.

**Lowest objectivity**: Implementation order (3/10) -- ordered by mathematical elegance, not user value or risk.

### Cost-Benefit Challenge

**Worst ROI decisions:**
- **Temporal Edge Validity (3/10 VFE)**: 3-4 days build + perpetual WHERE clause discipline + unbounded dead-edge storage growth. At 146 edges and 28 days of history, the edge history is worth nothing.
- **3-tier Profiles (2/10 VFE)**: Configuration theater. One user, one set of defaults.
- **Forgetting Curve (4/10 VFE)**: 3 days of work for a system that will produce near-uniform output for the next 2-3 months (all files are recent).

**Best ROI path**: HC's simpler alternative (use existing `summary` fields in long-range queries, 2 hours) + IR's search_log table (1 day) = measurement + token reduction groundwork without building anything premature.

### Alternative Paths

**Best unexplored path**: **Capture-time schema enforcement + real-time semantic deduplication.** Instead of 5 post-hoc systems (forgetting, temporal edges, clustering, summaries, routing weights) to compensate for unstructured input, enforce lightweight structure at the moment of capture: a Telegram quick-reply keyboard that takes 2 seconds to tag dimension and intent, plus an immediate cosine-similarity check surfacing "you've written about this 3 times" before the note is saved. The entire 18-day plan addresses symptoms of "captures lack structure"; the root cause fix is 2-3 days.

**Other missed alternatives:**
- **Inline entity extraction**: Add an `entities` field to the existing LLM classification call's response schema. Zero new infrastructure.
- **Rolling memo instead of 3-tier consolidation**: One Claude call/day producing 200-token structured memo appended to a single file. Zero schema migration.
- **Contextual bandits for routing weights**: Thompson Sampling over 4 channels auto-adjusts per command based on observed engagement. ~30 lines, no static archetypes.

### Risk Amplification

**Top 3 unidentified risks:**

1. **Silent Correctness Failure Accumulation**: Every addition can fail silently -- FC decay runs but recall events aren't recorded; TE validity updates run but a missed WHERE clause returns stale edges; HC summaries contain malformed JSON with silently dropped fields. No ground-truth validation loop exists anywhere. The system has 826 tests for "does it run?" and zero tests for "is the output correct over time?"

2. **Single Human-in-the-Loop Bottleneck**: 5 new attention-demanding features on top of 7 existing alert types. Concept graduation approvals, fading memory reviews, alert dismissals, summary quality checks -- all flow through one person. The PKM system designed to reduce cognitive overhead risks becoming a second job demanding daily review of its own outputs.

3. **Provenance Erasure**: Weekly/monthly summaries are written as real vault files, indexed and embedded with no distinction from user-authored content. Over 12-18 months, LLM-generated content progressively outweighs user content in semantic search. The `ghost` command begins impersonating Claude's interpretation of the user rather than the user themselves. There is no content-type weighting, no provenance flag, and no design principle preventing this outcome.

**Critical specific risks:**
- SQLite WAL contention during nightly job chain (9 jobs, 2:30am-7:00am, all writing to same DB)
- PTB JobQueue uses wall-clock scheduling, not dependency chains -- jobs can collide if one overruns
- DBSCAN cluster staleness: cluster computed at 2:30am, user approves at 9am, vault has changed
- macOS Time Machine/Spotlight opening the DB file during maintenance window can cause SQLITE_BUSY

## Blind Spots Exposed

These issues were found by only 1-2 lenses, making them the most valuable findings:

1. **Provenance erasure from LLM summaries** (Risk only) -- No mechanism distinguishes user-authored from LLM-generated content in the vault. Over time, Claude's summaries dominate the knowledge graph.

2. **Ghost voice degradation** (Risk only) -- Structured extraction summaries make the digital twin increasingly clinical and detached from the user's authentic reasoning style.

3. **Capture-time schema enforcement** (Alternatives only) -- A 2-3 day root-cause fix (Telegram quick-reply tagging at capture time) could eliminate the need for most of the 18-day plan.

4. **PTB JobQueue collision hazard** (Risk + Feasibility) -- Jobs scheduled by wall-clock time, not dependency chains. Overruns cause parallel writes to SQLite.

5. **DBSCAN approval staleness** (Risk only) -- Clusters computed at 2:30am are stale by 9am when the user approves. No revalidation at approval time.

6. **Missing embedding pipeline for captures_log** (Feasibility only) -- Concept Graduation assumes capture embeddings exist. They don't. This is an unacknowledged prerequisite.

## Final Verdict

**Would a staff engineer approve this?**

### APPROVE WITH REVISIONS

**What's strong enough to keep:**
- Stay on SQLite (6.7 avg) -- correct architectural call, well-justified
- Defer quarterly consolidation and community expansion (7.2 avg) -- right instinct, apply it more broadly
- Advisory-only forgetting (6.5 avg) -- correct guardrail for irreplaceable personal data
- Concept Graduation core idea (4.7 avg, 7/10 user impact) -- the only addition that closes a real behavioral loop today

**What must change before proceeding:**

1. **Invert the implementation order.** Ship Concept Graduation first (Week 1) -- it's the only addition with high user impact (7/10). Use simplified detection (existing community detection or keyword matching) instead of DBSCAN at 52 docs. Add IR's search_log table (1 day) for measurement infrastructure. Defer FC, TE, and HC until 90+ days of journal data exist.

2. **Kill the 3-tier profile system.** Hard-code one set of well-commented defaults. Add configurability only when a user actually requests it. This is a single-user tool.

3. **Add provenance flags before shipping any LLM-generated content to the vault.** A `source: system` frontmatter field and a `node_type: summary` in vault_nodes. Weight user-authored content higher than LLM-generated content in all search channels. This prevents the provenance erasure risk identified by the Risk Amplifier.

4. **Address the nightly job chain write-lock contention.** The 5 new additions create 9+ nightly jobs writing to the same SQLite file between 2:30am-7:00am. Add dependency-chain scheduling (each job triggers the next on completion) rather than wall-clock scheduling. Add a mutex or queue to prevent parallel writes.

5. **Add integration tests for cross-addition interactions.** The document has no test plan for the combined 5-addition system. The emergent behavior of FC decay + TE validity + HC summarization + CG clustering is entirely untested.

**What should be reconsidered entirely:**

- **Forgetting Curve (4.1 avg)**: The Ebbinghaus model is a category error -- rote memorization dynamics don't apply to knowledge retrieval. A simple SQL query (`WHERE last_modified < date('now', '-30 days')`) achieves the "surface fading content" use case in 5 lines. Revisit at 200+ documents if the simple version proves insufficient.

- **Temporal Edge Validity (4.0 avg)**: Massive complexity (130 LOC + 9-file WHERE clause updates + edge_episodes table) for near-zero current value on a 28-day graph. Add `created_at` timestamps to edges (2 lines) now. Build the full temporal layer in 3 months if /trace users request historical queries.

- **Hierarchical Consolidation (4.1 avg)**: The token savings argument obscures that the root problem is naive context loading, not context volume. Try loading only last-14-days of raw entries for long-range commands first. If that's insufficient, build the rolling-memo alternative (1 file, 1 daily Claude call) before committing to a 3-tier pipeline.
