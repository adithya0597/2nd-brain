# Grill Report: Week 4+ Conditional Feature Evaluation via Synthetic Data

**Target**: Claude's Week 4+ evaluation plan (inline document)
**Date**: 2026-03-29
**Griller team**: 7 independent adversarial agents (zero shared context)

## Executive Summary

The plan's **conclusions are probably correct** (implement Routing, defer the rest) but the **methodology is dangerous and unnecessary**. All 7 reviewers independently converged on the same core finding: this plan builds elaborate evaluation infrastructure to confirm a decision that was already made, using synthetic data that would contaminate the production database. The Risk Amplifier found a **P0 showstopper**: if the bot is live when 90 synthetic journal entries exist in the DB, the 10PM Notion sync will push all 90 fabricated entries to the Notion workspace -- irreversibly. The Feasibility Auditor found 3 critical blockers: hardcoded node IDs that will fail silently, `verified_at` never written in production (a code bug, not a data problem), and DBSCAN being the wrong target (graduation already uses SQL GROUP BY). The Cost-Benefit Challenger reduced the entire plan to: "Write one paragraph in HANDOFF.md documenting the 98.5% disagreement. Total time: 10 minutes."

## Per-Lens Critiques

### Devil's Advocate (Overall: 3/10)
- **Jaccard on sets of size 3 is statistically meaningless.** With top-3 results, Jaccard can only take 5 discrete values. The 0.25 threshold sits between 0.2 (0 shared) and 0.33 (1 shared). 98.5% "disagreement" is an artifact of comparing structurally different search channels, not evidence of a routing problem.
- **Channel disagreement is by design.** FTS5, vector, chunks, and graph SHOULD return different results -- that's the entire rationale for RRF fusion. Disagreement is a feature, not a bug.
- **Synthetic data for a personal system is especially suspect.** With 52 real documents, you have the complete dataset. Any feature that requires synthetic data to trigger has not been triggered.

### Feasibility Audit (Overall: 5/10)
**3 critical blockers found:**
1. **Hardcoded node IDs are wrong.** `gen_vault_edges()` uses IDs like 1621, 1636-1655. Actual vault_nodes has 67 rows (IDs ~1-67). FK violations cause silent INSERT failures via `INSERT OR IGNORE`.
2. **`verified_at` is never written.** `graph_ops.py` has the SQL but all 146 edges have NULL verified_at. The Temporal NO-GO is correct but because of a code bug, not because temporal edges lack value.
3. **DBSCAN is the wrong target.** `graduation_detector.py` explicitly says "NO DBSCAN, NO scikit-learn" -- it uses SQL frequency counting on `dimensions_json`. The feature already has 4 proposals (2 approved, 1 rejected, 1 snoozed). Kill criterion already met.

### Bias Detection (Overall: 3/10)
**Predetermined Conclusion Bias (1/10)**: The plan pre-declares "Expected outcome: Routing GO, Temporal NO-GO, Consolidation NO-GO, DBSCAN BORDERLINE" before any evaluation runs. The developer generates the data, writes the criteria, sets the thresholds, and predicts the outcome. This is not evaluation -- it is rationalization dressed as measurement.

**Threshold Gerrymandering**: Both key thresholds (Jaccard < 0.25, tokens > 30K) are set by the developer with no external benchmark, sensitivity analysis, or empirical calibration. Adjust either threshold by 20% and the verdict changes.

### Alternative Paths
**All 4 verdicts derivable from 5 SQL queries in under 5 seconds:**
```sql
-- Routing: Do channels disagree?
SELECT command, channels_used, COUNT(*) FROM search_log GROUP BY command, channels_used;
-- Temporal: Are edges stale?
SELECT COUNT(*) FROM vault_edges WHERE verified_at < date('now', '-30 days');
-- Consolidation: Is context too large?
SELECT AVG(LENGTH(content)) FROM journal_entries WHERE date >= date('now', '-60 days');
-- Graduation: Are captures accumulating?
SELECT COUNT(*) FROM captures_log WHERE date(created_at) < date('now', '-14 days');
-- Graduation: Is detector working?
SELECT status, COUNT(*) FROM concept_metadata GROUP BY status;
```

**Best unexplored path**: Fix `/trace` temporal ordering (sort results by date -- ~5 lines). This is the only broken user-visible behavior and it's being ignored in favor of evaluation infrastructure.

### Risk Amplification

**P0 SHOWSTOPPER: Notion sync will push synthetic data.**
If the bot is live when 90 synthetic journal entries exist, `job_notion_sync` at 10PM calls `get_unsynced_journal_entries()` which selects ALL entries not in `vault_sync_log`. All 90 synthetic entries qualify. They get pushed to Notion Notes DB. There is no delete path in the sync engine. **This permanently and irreversibly corrupts the Notion workspace.**

**Other critical risks:**
- **Community ID corruption persists after edge cleanup.** Deleting 500 synthetic edges does not restore `community_id` on vault_nodes. Stays wrong until bot restart.
- **Engagement metrics permanently inflated.** Staggered timestamps on synthetic edges inflate `engagement_daily.edges_created` for every date in the range.
- **UNIQUE(date) on journal_entries.** 90 synthetic entries must avoid all 30 real dates. No collision avoidance specified. Partial batch failure leaves inconsistent state.
- **WAL corruption on live restore.** "Restore from backup" while bot is running orphans the WAL file, corrupting the DB.
- **Alert fingerprints permanently suppressed.** MD5 hashes in `alerts.fingerprint` record as dismissed; real alerts won't re-fire for those patterns.

### User Impact Assessment (Overall: 1.6/10)
**The real user problem being ignored:** `/trace` returns graph-connected files but not in chronological order. The user gets a scatter of related notes, not an evolution timeline. This is fixable in ~5 lines (sort by date). Instead, dev time goes to evaluation infrastructure that produces no user-visible output.

**The 65 search_log entries are synthetic**, not real usage (queries like "morning routine", "transformer architecture" -- not how a real user searches their own PKM). The routing evaluation is measuring behavior on fabricated queries.

## Final Verdict

**Would a staff engineer approve this?**

### REJECT THE METHODOLOGY. APPROVE THE CONCLUSIONS.

**The conclusions are correct:**
- Routing: Ship it (25 lines, low risk, reversible)
- Temporal Edges: Defer (but fix the `verified_at` code bug)
- Consolidation: Defer (16K tokens is fine)
- DBSCAN: Moot (graduation already works and met its criterion)

**The methodology is rejected because:**
1. **It will corrupt production data.** Synthetic data injected into a live system with nightly jobs, Notion sync, and alert fingerprinting creates irreversible contamination across 5+ tables and an external system (Notion).
2. **It confirms what's already known.** The "Expected Outcome" section pre-declares the verdicts. The evaluation cannot falsify the hypothesis.
3. **It costs orders of magnitude more than the alternative.** 3 new scripts + 5 agents + synthetic data generation vs. 5 SQL queries + one paragraph in HANDOFF.md.
4. **The Jaccard metric is statistically invalid** for sets of size 3, and channel disagreement is by design, not evidence of a routing problem.

**What to do instead (30 minutes total):**
1. Run the 5 SQL queries from the Alternatives section against the live DB
2. Ship the 25-line routing change directly (it's low-risk and reversible)
3. Fix `/trace` temporal ordering (sort by date, ~5 lines) -- the only broken user-visible behavior
4. Fix the `verified_at` code bug in `graph_ops.py` (production code never writes it)
5. Write one paragraph in HANDOFF.md documenting these decisions and why
