# Grill Report: Intelligence Layer Proposal

**Target**: Intelligence Layer proposal for 2nd Brain capture pipeline
**Date**: 2026-04-01
**Griller team**: 7 independent adversarial agents (zero shared context)

## Executive Summary

The core diagnosis is correct: the capture pipeline IS the weakest link, and adding structured understanding would be transformative. **However, the proposal wraps a $300 steak in $3,000 of garnish.** The LLM extraction + Notion write-back + morning deadline reminders could ship in days and would immediately transform daily usage. But the proposal bundles it with a webapp, four classification axes, a second LLM vendor, and a full reminder system -- inflating a focused fix into a multi-month project. The most dangerous pattern: every decision defaults to the most complex option without asking "what is the simplest version that works?" A recurring "acquiescence bias" from the interview (user said "yes" to everything) produced a wish list, not a prioritized backlog.

## Per-Lens Critiques

### Devil's Advocate
**Strongest challenges:** The proposal abandons the existing "cheap first, expensive last" classifier philosophy by making LLM calls mandatory. Four axes create a combinatorial explosion with no conflict resolution logic described. The reminder system (3 channels) will cause notification fatigue -- 5 uncompleted tasks/day means 35 recurring reminders within a week. The "PhD library, kindergarten intake" metaphor is vivid but never substantiated with error rates or misclassification data.

**Weakest decisions flagged:** Gemini Flash (3/10) -- adding a second LLM provider for $7/year savings is indefensible. People split (4/10) -- two-tier storage creates retrieval gaps when archive flags exist. Four axes (4/10) -- adding Intent alone captures most value at a fraction of complexity.

### Feasibility Audit
**Grounded in code analysis.** The `ai_client.py` already has a Gemini backend (lines 64-132), making Decision 3 near-zero effort (9/10). But the webapp (3/10) is a multi-month project: no API layer exists, no auth system, no hosting plan, no design system. The current system has zero JavaScript. The Notion People integration already exists (10/10) -- no new work needed.

**Key insight:** Run LLM extraction only when `is_actionable=True` (existing regex at classifier.py:71-75). This covers the high-value case without adding latency to reflections. At ~5 actionable captures/day, cost drops to $0.005/day.

### Bias Detection
**Three systemic biases identified:**
1. **Acquiescence bias** from the interview -- asking "would you like X?" and treating every "yes" as a validated requirement. No trade-offs were forced.
2. **Complexity bias** -- every decision defaults to the sophisticated option (4 axes over 1, 3 layers over 2, 3 reminder channels over 1).
3. **Sunk cost / IKEA effect** -- the substantial existing infrastructure (830 tests, 24+ tables, graph pipeline) creates pressure to keep building rather than asking "is the current system being used daily?"

**Critical gap:** No usage data, error rates, or user satisfaction metrics for the existing system are presented anywhere in the proposal.

### Cost-Benefit Challenge
**1-year TCO estimates:**
- LLM per capture: ~$1,200-1,800 (engineering time, not API cost)
- Four axes: ~$4,200 (schema migration + full pipeline integration)
- Three-layer architecture: ~$12,000-22,600 (120-226 hours + hosting)
- Next.js over Streamlit: ~$4,500-9,000 additional vs Streamlit path
- People automation: ~$3,000 to save 3.5 hrs/year of manual entry
- Full reminder system: ~$5,400 to replicate Apple Reminders poorly

**Fundamental tension:** The proposal conflates two goals -- "make my personal tool better" (low cost, high ROI) and "build a demo-able product" (high cost, uncertain ROI). These have fundamentally different cost structures.

### Alternative Paths
**Best unexplored path:** Skip the LLM extraction pipeline entirely. Use pure retrieval against existing data. Embed the capture, cosine-match against cached Notion project/people names from `notion-registry.json`. Add a `reminder_date` column and a 20-line date regex. Include reminders in the existing morning briefing. ~150 lines of new code, zero new dependencies, zero API cost. Solves the top 3 complaints in an afternoon.

**Other missed alternatives:**
- **Local extraction** via spaCy NER (~5ms/message) + intent reference embeddings (same pattern as existing zero-shot tier)
- **Telegram Mini Apps** -- embed a webapp inside Telegram itself, collapsing two layers into one
- **FastAPI + HTMX** instead of Next.js -- reuse existing Python modules directly, no second language
- **Google Calendar API** for reminders (MCP tools already available in the environment)
- **LiteLLM** for provider-agnostic LLM access, making the Gemini vs Haiku decision reversible at zero code cost

### Risk Amplification
**Top 3 unidentified risks:**
1. **SQLite as backbone for 3 concurrent consumers** -- WAL mode allows concurrent reads but ONE writer. Bot + webapp API server + scheduled jobs competing for writes will produce `SQLITE_BUSY` errors under real usage. No mitigation discussed.
2. **Single point of failure: the developer** -- No runbook, no disaster recovery plan, no "bus factor" mitigation. Adding more external dependencies (Google API, hosting provider, Node.js runtime) increases the blast radius.
3. **Feature creep masquerading as architecture** -- Each addition is individually reasonable. Together, they double the system's complexity. The system meant to reduce cognitive load becomes itself a source of cognitive load.

**Privacy risk flagged:** Adding Gemini Flash means sending every personal thought, financial note, health entry, and relationship reflection to TWO AI vendors instead of one, with different data retention policies.

### User Impact Assessment
**The core feature IS user-justified (9/10).** The gap between "classified to Mind & Growth" and "Notion task created with due date, linked person, and reminder" is enormous. This is the feature that turns the bot from a note-taker into an assistant.

**But most surrounding decisions are invisible to the user:**
- Four classification axes: 4/10 (user cares about what the bot DOES, not how it classifies)
- Gemini vs Haiku: 2/10 (user literally cannot tell)
- People storage location: 2/10 (user cares about the link existing, not where the record lives)
- Next.js vs Streamlit: 3/10 (cosmetic; the intelligence IS the demo)

**Best verdict quote:** "The most impressive demo is not a pretty UI -- it is sending a stream-of-consciousness Telegram message and watching a Notion task appear with the right due date, person, and project 2 seconds later. That demo does not need a webapp at all."

## Blind Spots Exposed

1. **No usage data for the existing system** -- flagged by Bias Detector alone. The proposal never presents how often the current classifier is wrong, how many captures per day are sent, or whether the morning briefing is even being read. Building on assumptions rather than data.
2. **SQLite concurrency under three consumers** -- flagged by Risk Amplifier alone. This is a structural risk that could cause mysterious failures under real usage.
3. **Privacy implications of dual-vendor AI** -- flagged by Risk Amplifier alone. Every personal thought now flows through two AI companies with different data policies.
4. **Multi-intent messages** -- flagged by Feasibility Auditor alone. "Had a great run (reflection), need new shoes (task), knee still hurts (health note)" -- the system has no way to split these.
5. **Alarm fatigue from recurring reminders** -- flagged by Devil's Advocate and Risk Amplifier. 5 uncompleted tasks/day = 35 recurring reminders within a week.

## Alternative Approaches Missed

1. **Retrieval-based linking (no LLM):** Cosine-match captures against cached project/people names from `notion-registry.json`. ~150 lines, zero API cost, solves top 3 complaints.
2. **Local NER + intent embeddings:** spaCy for entities (~5ms), zero-shot intent via reference embeddings (same pattern as existing ICOR classifier). 80% of LLM quality at $0 cost.
3. **Telegram Mini Apps for the management UI:** Embed the webapp inside Telegram itself. One deployment, one auth system, one notification channel.

## Final Verdict

**APPROVE WITH MAJOR REVISIONS**

### What must change before proceeding:

1. **Drop Gemini Flash.** Use Haiku (already integrated) or the existing Gemini backend in `ai_client.py`. Do not add a second LLM vendor for a $7/year saving at $1,100 engineering cost.

2. **Gate LLM extraction on `is_actionable`.** Don't call the LLM on every capture. The existing regex already identifies ~70% of actionable messages. Run extraction only on those. Reflections and article shares don't need entity extraction.

3. **Start with ONE new axis: Intent.** Add Intent (task/update/idea/reflection) only. Entity and Action extraction are downstream consequences of Intent, not parallel axes. Reduce four simultaneous classification changes to one.

4. **Kill the Next.js webapp.** Zero JavaScript expertise in the codebase. If a demo UI is needed, Streamlit (2-3 days) or even a Notion dashboard page (0 code). The demo is the intelligence, not the UI framework.

5. **Simplify reminders to one line of code.** Add a `due_date` column to `action_items`. Add one SQL query to the morning briefing: `WHERE due_date = date('now')`. That IS the reminder system. Telegram already pushes notifications.

### What is strong enough to keep:

- **The core diagnosis** -- the capture pipeline IS the weakest link and needs structured understanding
- **The sequential priority** -- fix the pipeline before building any UI
- **People in Notion** -- already works, no change needed
- **Notion as the task/project backend** -- correct, don't duplicate it

### What should be reconsidered entirely:

- **Three-layer architecture** -- the existing Telegram + Notion is two layers that already struggle with sync (P1 TOCTOU race). Adding a third layer before fixing the two-layer sync is building on a cracked foundation.
- **The "demo-able product" goal** -- this conflicts with "personal productivity tool." Pick one. The most impressive demo requires zero webapp: live-text a capture into Telegram, watch a structured Notion task appear in 2 seconds. Record a Loom.
- **The scope itself** -- the proposal is 8 decisions, 4 new systems, 2 new languages. The fix is 1 decision: add structured extraction to actionable captures and write the results to Notion properly.
